### 问题 A.2 🔴 `save_db_config` 直接写入 `.env`，无转义 + dead code

**位置**：`api/pipeline/service.py` — `save_db_config()`

**现状代码**：

```python
def save_db_config(self, config: Dict[str, Any]) -> Dict[str, Any]:
    env_path = ".env"
    ...
    for key in db_keys:
        value = config.get(key, "")
        new_lines.append(f"{key.upper()}={value}\n")  # ← value 完全未校验/未转义
    ...
    with open(env_path, "w", encoding="utf-8") as f:
        f.writelines(env_lines)
```

**问题分析**：

三个问题叠加：

1. **无转义**：`db_password` 等字段直接写入，**不带引号、不转义**。如果密码含 `#` 或换行，`.env` 解析会错乱
2. **无备份**：没有备份原 `.env`，写错了无法回滚
3. **dead code**：同时存在两个写入循环（`for key in db_keys: new_lines.append(...)` 和后面 `for i, line ... updated_keys.add(key)`），**逻辑重复且 `new_lines` 实际从未被使用**

**建议方案**：

```python
# 密码字段用双引号包裹
for key in db_keys:
    value = config.get(key, "")
    if key == "db_password":
        new_lines.append(f'{key.upper()}="{value}"\n')  # 引号包裹
    else:
        new_lines.append(f"{key.upper()}={value}\n")

# 写入前备份
import shutil
if os.path.exists(env_path):
    shutil.copy(env_path, f"{env_path}.bak")

# 删除未使用的 new_lines 块
```

**优先级**：🔴 P0
**工作量**：30 分钟
**收益**：安全 + 防止配置损坏

### 问题 A.3 🔴 `estimate_tokens` 中文系数偏低，导致上下文超限

**位置**：`rag/prompt_template.py:178` + `rag/memory.py:11`（两处重复定义）

**现状代码**（两个文件**完全相同**）：

```python
def estimate_tokens(text: str) -> int:
    cn_chars = len(re.findall(r'[\u4e00-\u9fff]', text))
    en_words = len(re.findall(r'[a-zA-Z]+', text))
    other_chars = len(re.findall(r'[^\u4e00-\u9fffa-zA-Z]+', text))
    return int(cn_chars * 1.5 + en_words * 1.3 + other_chars * 0.5)
```

**问题分析**：

- **中文实际是 ~1.7-2.0 token/字**（GPT 系列、Claude、GLM 都是这样），用 1.5 会**低估 13-33%**
- 后果：`max_context_tokens=3000` 实际可能塞进 3500-4000 token，触发模型截断或报错
- **DRY 违规**：两个文件重复定义同一个函数

**建议方案**：

1. 系数改成 `cn_chars * 2.0`（保守估计更安全）
2. 抽到 `utils/tokens.py`，两边 import

```python
# utils/tokens.py
def estimate_tokens(text: str) -> int:
    cn_chars = len(re.findall(r'[\u4e00-\u9fff]', text))
    en_words = len(re.findall(r'[a-zA-Z]+', text))
    other_chars = len(re.findall(r'[^\u4e00-\u9fffa-zA-Z]+', text))
    return int(cn_chars * 2.0 + en_words * 1.3 + other_chars * 0.5)
```

**优先级**：🔴 P0
**工作量**：10 分钟
**收益**：避免上下文超限导致的 LLM 报错

## 二、P1 高优先级问题

### 问题 B.1 🔴 切片是字符级硬切，中文常切断句子

**位置**：`embeddings/chunker.py` — `chunk()` 和 `chunk_with_pages()`

**现状代码**：

```python
def chunk(self, text: str, source: str = "unknown") -> List[Dict]:
    while start < len(text):
        end = start + self.chunk_size  # 500
        content = text[start:end]      # ← 纯字符切片，不管语义
        ...
        start += self.chunk_size - self.chunk_overlap
```

**问题分析**：

- 500 字符在中文里约 250-300 字，**会从句子中间切断**
- 切断的句子既影响 embedding 质量（语义不完整），又影响 BM25（关键词被切到两个 chunk）
- 这是检索召回率的**隐形杀手**——前面 RRF 调得再好，切片烂也白搭

**建议方案**：

在字符切片基础上加**句子边界对齐**：

```python
def chunk(self, text, source="unknown"):
    ...
    while start < len(text):
        end = start + self.chunk_size
        # 优先在句号/换行处切
        if end < len(text):
            for sep in ['\n', '。', '！', '？', '.', '!', '?', '；']:
                last_sep = text.rfind(sep, start, end)
                if last_sep > start + self.chunk_size // 2:  # 至少切一半
                    end = last_sep + 1
                    break
        content = text[start:end]
        ...
```

**优先级**：🔴 P1
**工作量**：30 分钟
**收益**：对检索质量提升立竿见影

**位置**：`embeddings/embedder.py` / `rag/router.py` / `rag/chain.py`（三处）

**现状代码**：

```python
# embedder.py
with httpx.Client(timeout=60) as client:

class EmbeddingClient:

        ...
        self._client = httpx.Client(timeout=60)
    
    def _call_api(self, texts):
        response = self._client.post(...)  # 复用连接池
```

**优先级**：🔴 P1
**工作量**：30 分钟
**收益**：高并发下延迟显著下降

### 问题 B.3 🔴 `_trim_by_tokens` 会切断 user/assistant 配对

**位置**：`rag/memory.py:_trim_by_tokens()`

**现状代码**：

```python
def _trim_by_tokens(self, session_id: str):
    ...
    for i in range(len(messages) - 1, -1, -1):
        msg_tokens = estimate_tokens(messages[i].get("content", ""))
        if total_tokens + msg_tokens > self.max_tokens:
            break  # ← 这里 break，但 keep_from 还没回退
        total_tokens += msg_tokens
        keep_from = i
    
    if keep_from > 0:
        self.histories[session_id] = messages[keep_from:]
```

**问题分析**：

token 超限时直接从 `keep_from` 截断，**可能正好砍在 user/assistant 中间**，留下一个孤立的 user 消息或 assistant 消息。OpenAI/Claude/GLM API 对消息角色顺序有要求（必须 user 在前），会导致请求失败。

**建议方案**：

截断后强制对齐到偶数边界（user 开头）：

```python
if keep_from > 0:
    # 确保从 user 消息开始（角色配对完整性）
    if messages[keep_from]["role"] != "user":
        keep_from += 1
    self.histories[session_id] = messages[keep_from:]
```

**优先级**：🔴 P1
**工作量**：10 分钟
**收益**：避免多轮对话 API 报错

**问题分析**：

**已确认可利用（2026-06-23 实测）**：

```python
# 攻击者发送：POST /api/chat {"query": "hello", "session_id": "../../test_traversal"}
# memory.py 实际执行：

| 测试输入 | 预期位置 | 实际位置 | 结果 |
|---------|---------|---------|------|
| `../../test_traversal` | `data/sessions/test_traversal.json` | `./test_traversal.json`（项目根） | ⚠️ **路径穿越成功** |

**利用场景**：

- **写入**：攻击者可向项目任意可写目录写入 JSON 文件（对话历史格式）
- **读取**：`GET /{session_id}/history` 的 session_id 在 URL 路径中，会被 Starlette 规范化，**不可利用**；但 `DELETE /{session_id}` 同理也不可利用

**建议方案**：

import re

    """校验 session_id，防止路径遍历"""
    if not re.match(r'^[a-zA-Z0-9_-]{1,64}$', session_id):
        raise ValueError(f"非法 session_id: {session_id}")
    return session_id

# 在 routes_chat.py 中使用
session_id = _validate_session_id(req.session_id) if req.session_id else str(uuid.uuid4())
```

或在路由层**完全不接受用户传入** session_id，只服务端生成。

**工作量**：10 分钟
**收益**：堵住已确认的路径遍历漏洞

### 问题 B.6 🔴 流式输出失败时记忆断片

**位置**：`rag/chain.py:_chat_stream()` 和 `_chat_stream_database()`

**现状代码**：

```python
def _chat_stream(self, query, messages, session_id, docs, query_type):
    ...
    full_answer = ""
    for chunk in self._call_llm(messages, stream=True):
        full_answer += chunk
        yield chunk
    
    self.memory.add(session_id, "assistant", full_answer)  # ← 只在生成器跑完才执行
```

**问题分析**：

如果客户端中途断开（关闭页面、网络中断），生成器抛 `GeneratorExit`，**最后一行 `memory.add` 不执行**。下次对话时记忆里只有用户的 query，没有 assistant 回答，多轮上下文断裂。

**建议方案**：

用 try/finally 保证入记忆：

```python
full_answer = ""
try:
    for chunk in self._call_llm(messages, stream=True):
        full_answer += chunk
        yield chunk
finally:
    if full_answer:  # 即使断开，已生成的部分也入记忆
        self.memory.add(session_id, "assistant", full_answer)
```

**优先级**：🔴 P1
**工作量**：10 分钟
**收益**：多轮对话上下文不丢失

### 问题 C.5 ⚠️ 后台任务用 `BackgroundTasks` 单进程，长任务阻塞后续请求

**位置**：`api/routes_pipeline.py:process_files()`

**现状代码**：

```python
background_tasks.add_task(run_background)  # ← FastAPI BackgroundTasks 是同步阻塞的
```

**问题分析**：

Spark 处理可能几分钟，期间这个 worker 被占住，其他请求排队。

**建议方案**：

换 `asyncio.create_task` 或 Celery。

**优先级**：🟡 P2
**工作量**：1 小时
**收益**：避免长任务拖垮 API 响应
