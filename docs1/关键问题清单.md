## 一、查询路由问题（router.py）

### 问题 1.1 ⚠️ database 关键词误判优先级过高

**位置**：`rag/router.py` — `_check_database_query()` 与 `route_query()`

**现状代码**：

```python
DB_QUERY_KEYWORDS = [
    '查询', '统计', '排名', '排行', '列表', '多少', '几个', '哪些',
    '最多', '最少', '最高', '最低', '平均', '总计', '总数', '数量',
    'top', '前', '后', '大于', '小于', '等于', '超过', '低于',
    '播放量', '弹幕', '时长', '发布', '视频',
]

# 路由顺序：rule_based_route → _check_database_query → _check_general_or_rag → llm_classify
```

**问题分析**：

路由顺序里 `_check_database_query` 排在 LLM 之前，且规则是**关键词命中 ≥2 个就判定为 database**。但关键词表过于宽泛，会导致严重的误判：

| 用户输入 | 命中关键词 | 实际意图 | 错误路由 |
|---------|-----------|---------|---------|
| "知识库里**查询**到的**统计**数据有哪些" | 查询、统计 | RAG | ❌ database |
| "LangChain 的核心组件**有哪些**" | 哪些（仅 1 个，侥幸） | RAG | 勉强正确 |
| "**查询**功能**排名**如何" | 查询、排名 | general/RAG | ❌ database |

**根本原因**：database 路由不应该和 RAG 路由并列，它应该是一个**强约束的窄通道**。柔性关键词匹配（`hit_count >= 2`）误判率太高。

**建议方案**：

1. **删除柔性匹配规则**（`hit_count >= 2` 整段删掉），只保留 `strong_patterns`
2. **增加数据库可用性前置判断**：未配置数据库时根本不进 database 路径

```python
def route_query(query: str) -> QueryType:
    # 0. 规则匹配（chitchat）
    if (result := rule_based_route(query)) is not None:
        return result
    
    # 1. database 检测：仅当数据库实际配置 + 强模式命中才走
    if settings.db_type:
        if _is_strong_database_query(query):  # 只用 strong_patterns
            return QueryType.DATABASE
    
    # 2. general 启发式
    if (result := _check_general_or_rag(query)) is not None:
        return result
    
    # 3. LLM 兜底（prompt 里已经教过 database 判定）
    return llm_classify(query)
```

**优先级**：🔴 P0
**工作量**：30 分钟
**收益**：消除严重误路由

### 问题 1.2 ⚠️ 四层路由命中即终判，无回查机制

**位置**：`rag/router.py` — `route_query()`

**现状代码**：每一层都是"命中就直接 return"，没有任何回查机制。

```python
def route_query(query: str) -> QueryType:
    result = rule_based_route(query)
    if result is not None:
        return result  # ← 命中即终判
    
    result = _check_database_query(query)
    if result is not None:
        return result  # ← 命中即终判
    ...
```

**问题分析**：

规则层的 `GREETING_PATTERNS` 和 `SMALL_TALK_PATTERNS` 使用 `^...$` 锚定整句，当前看似安全，但存在脆弱性：

```python
GREETING_PATTERNS = [
    r'^(你是谁|你叫什么|你能做什么|介绍一下你自己|你是什么)[\s?？。.]*$',
]
```

- 用户问 "你是谁开发的" → 不匹配（有"开发的"），正确
- 但如果模式稍微扩张（比如加上 ".*"），就会误命中 chitchat 且**无法回退**

**根本原因**：规则层承担了过重的判断责任。chitchat 这种语义高度确定的可以走规则，但 general/database 这种语义复杂的应该交给 LLM。

**建议方案**：

把规则层收缩到**只处理 GREETING 和 SMALL_TALK**，删掉 `GENERAL_PATTERNS` 和 DB 关键词层：

- 规则层只管零成本拦截问候（占流量大头）
- 剩下的全部走 LLM，准确率从 ~85% → ~95%+
- 多花的 LLM 成本可接受（分类 prompt 仅 ~300 token，max_tokens=15）

**优先级**：🟢 P2
**工作量**：重构（半天）
**收益**：长期可维护性

### 问题 1.3 🔴 LLM 分类失败默认走 RAG，导致闲聊被拒答

**位置**：`rag/router.py` — `llm_classify()`

**现状代码**：

```python
def llm_classify(query: str) -> QueryType:
    try:
        ...
    except Exception as e:
        logger.warning(f"LLM 分类失败，默认走 RAG: {e}")
    
    return QueryType.RAG  # 分类失败时默认走 RAG
```

**问题分析**：

LLM 一旦超时/限流，**所有查询都走 RAG**。但 RAG 路径里有硬校验（`chain.py` 里 `if not docs: 返回拒答`），后果链：

```
用户："你好"
  → LLM 分类超时
  → 默认 RAG
  → 向量检索 "你好"
  → 检索不到相关文档
  → 返回 "抱歉，知识库中没有找到与您问题相关的内容"
```

**用户体验灾难**：问候被当成知识库问题拒答。

**建议方案**：失败时默认走 `general`（纯 LLM 兜底，至少能回答）：

```python
except Exception as e:
    logger.warning(f"LLM 分类失败，回退 general: {e}")
    return QueryType.GENERAL  # ← 改这里
```

**优先级**：🔴 P0
**工作量**：1 行代码
**收益**：立竿见影，避免用户体验灾难

### 问题 1.4 ⚠️ LLM prompt 与路由层规则存在职责重叠

**位置**：`rag/router.py` — `CLASSIFY_PROMPT` vs 规则层

**现状代码**：

`CLASSIFY_PROMPT` 已经教过 LLM 如何判断 database：
```
database — 需要查询数据库中的结构化数据才能回答，涉及统计、排序、筛选、排名等操作
```

但 `_check_database_query` 又用关键词规则做了一遍 database 判断，**两套机制职责重叠**，且规则层的准确率明显低于 LLM。

**问题分析**：

- 规则层判断 database：靠 `sum(1 for kw in DB_QUERY_KEYWORDS if kw in query_lower) >= 2`，纯字面匹配
- LLM 判断 database：理解语义，准确率高得多
- **结果**：低准确率的规则层抢在 LLM 之前做判断，把 LLM 的判断空间挤掉了

**建议方案**：

database 这种低频但需要语义理解的路径，**完全交给 LLM**。规则层只保留零成本拦截 chitchat 的职责。

**优先级**：🟡 P1
**工作量**：30 分钟（配合问题 1.1 一起改）
**收益**：路由准确率提升

## 二、检索逻辑问题（retriever.py）

### 问题 2.1 🔴 BM25 缓存全量加载 + 每次插入都重建

**位置**：`rag/retriever.py` — `_update_bm25_cache()` 与 `insert_documents()`

**现状代码**：

```python
def _update_bm25_cache(self):
    """更新 BM25 检索缓存（分批加载，突破单次查询限制）"""
    all_results = []
    offset = 0
    batch_size = 10000
    while True:
        batch = self._client.query(
            self.collection_name,
            filter='chunk_id != ""',
            output_fields=["content", "filename", "chunk_id", "page_number"],
            limit=batch_size,
            offset=offset
        )
        ...
    self._bm25_docs = all_results

def insert_documents(self, chunks: List[Dict]) -> int:
    ...
    self._update_bm25_cache()  # ← 插一条就全量重查
    return count
```

**问题分析**：

两个问题叠加：

**(a) 全量加载内存爆炸**：BM25 要求把整个语料库的 token 列表常驻内存：

```python
corpus_tokens = [list(jieba.cut(doc.get("content", ""))) for doc in self._bm25_docs]
```

当前 53 个切片没问题，但**文档到 5000 切片时，每次查询都要重新切分整个语料库**——jieba 分词 5000 段文本，单次查询可能要 1-2 秒。

**(b) 每次插入都全量重建**：插入一条就触发 `_update_bm25_cache()`，把整个 collection 重新 query 一遍。

**建议方案**：增量更新 + 分词结果缓存：

```python
class VectorRetriever:
    def __init__(self, ...):
        ...
        self._bm25: Optional[BM25Okapi] = None
        self._token_cache: Dict[str, List[str]] = {}  # chunk_id -> tokens
    
    def _update_bm25_cache(self):
        """只对新增/变更的 chunk 重新分词，复用已缓存的分词结果"""
        docs = self._fetch_all_docs()
        for d in docs:
            if d["chunk_id"] not in self._token_cache:
                self._token_cache[d["chunk_id"]] = list(jieba.cut(d["content"]))
        self._bm25_docs = docs
        self._bm25 = BM25Okapi([self._token_cache[d["chunk_id"]] for d in docs])
```

更彻底的方案：**用 Milvus 2.4+ 的全文检索功能**（scattered/scalar 字段 + BM25 函数），让数据库自己做倒排索引，省掉内存缓存。

**优先级**：🟢 P2
**工作量**：2 小时
**收益**：规模化必备（当前规模无影响，扩展时关键）

### 问题 2.4 🔴 BM25 每次查询都重新实例化 BM25Okapi — 性能炸弹

**位置**：`rag/retriever.py:192-206`（`_bm25_search()` 函数内部）

**现状代码**：

```python
def _bm25_search(self, query: str, top_k: int = 5) -> List[Dict]:
    """BM25 关键词检索"""
    if not self._bm25_docs:
        return []
    
    try:
        import jieba
        from rank_bm25 import BM25Okapi
        
        # 分词
        query_tokens = list(jieba.cut(query))
        corpus_tokens = [list(jieba.cut(doc.get("content", ""))) for doc in self._bm25_docs]  # ← 每次查询都重新分词整个语料库
        
        # BM25 检索
        bm25 = BM25Okapi(corpus_tokens)  # ← 每次查询都重建 BM25 索引
        scores = bm25.get_scores(query_tokens)
        ...
```

**问题分析**：

这是**整个检索模块最严重的性能问题**。`_bm25_search()` 每次被调用时，在函数内部做两件与语料规模线性相关的昂贵操作：

1. **对整个语料库重新 jieba 分词**（`corpus_tokens = [list(jieba.cut(...)) for doc in self._bm25_docs]`）
2. **重新构造 `BM25Okapi` 实例**（涉及 IDF 表计算、词频统计）

这两步的开销**与查询内容无关，只与语料规模有关**——它们的结果对所有查询都是相同的，本来应该在数据变更时算一次缓存起来。但当前实现把它们塞进了每次查询的热路径里。

**已验证的真实性能影响**（用项目 conda 环境 `kbqa` 跑基准测试，5 查询/轮 × 5 轮取平均）：

| 语料规模 | 当前实现（ms/查询） | 优化后（ms/查询） | 加速比 |
|---------|-------------------|------------------|--------|
| **53 切片（当前知识库）** | **98.93** | 0.13 | **783x** |
| 500 切片 | 446.34 | 0.82 | 544x |
| 2000 切片 | 1744.02 | 3.10 | 563x |
| 5000 切片 | **4242.95（4.2 秒）** | 6.52 | 651x |

**结论**：当前 53 切片规模下，每次 BM25 查询就要 **~100ms**（用户已能感知卡顿）；到 5000 切片会变成 **4.2 秒**——这是让整个问答系统看起来"卡死"的性能瓶颈。

**根因诊断**：

`_update_bm25_cache()`（line 59-84）已经缓存了文档列表 `self._bm25_docs`，但**只缓存了原始文档，没缓存分词结果和 BM25 实例**。分词和建索引这两步本应在 `_update_bm25_cache()` 里完成，却错误地被放进了每次查询的 `_bm25_search()` 里。

**建议方案**：分词结果缓存 + BM25 实例复用：

```python
class VectorRetriever:
    def __init__(self, embedder=None):
        ...
        self._bm25_docs: List[Dict] = []
        self._token_cache: Dict[str, List[str]] = {}  # 新增：chunk_id -> tokens
        self._bm25: Optional["BM25Okapi"] = None      # 新增：复用的 BM25 实例
        self._connect()
        self._ensure_collection()
        self._update_bm25_cache()
    
    def _update_bm25_cache(self):
        """更新 BM25 缓存（文档 + 分词 + BM25 实例一次性构建）"""
        # ... 原有的文档加载逻辑，得到 self._bm25_docs ...
        
        # 增量分词：只对未缓存的新 chunk 重新分词
        for d in self._bm25_docs:
            cid = d.get("chunk_id", "")
            if cid not in self._token_cache:
                self._token_cache[cid] = list(jieba.cut(d.get("content", "")))
        
        # 一次性构造 BM25 实例（数据变更时才算一次）
        corpus_tokens = [self._token_cache[d.get("chunk_id", "")] for d in self._bm25_docs]
        self._bm25 = BM25Okapi(corpus_tokens)
    
    def _bm25_search(self, query: str, top_k: int = 5) -> List[Dict]:
        """BM25 关键词检索（只做查询相关的工作）"""
        if not self._bm25_docs or self._bm25 is None:
            return []
        
        query_tokens = list(jieba.cut(query))  # 只分词查询，不动语料
        scores = self._bm25.get_scores(query_tokens)  # 复用 BM25 实例
        ...
```

数据变更（`insert_documents` / `delete_by_filename`）时调用 `_update_bm25_cache()` 重建即可，已经在做了，无需改动调用点。

**优先级**：🔴 P0
**工作量**：1 小时（含验证）
**收益**：检索延迟从 ~100ms 降到 <1ms（当前规模），规模化时避免 4 秒级卡顿

## 三、评分排序问题（RRF 融合）

### 问题 3.1 ⚠️ RRF k=60 对小规模知识库偏大

**位置**：`config/settings.py:45` + `rag/retriever.py:_rrf_fusion()`

**现状代码**：

```python
# settings.py
rrf_k: int = 60  # RRF (Reciprocal Rank Fusion) 参数

# retriever.py
def _rrf_fusion(self, vector_results, keyword_results, k=None):
    k = k or settings.rrf_k
    for rank, doc in enumerate(vector_results):
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
```

**问题分析**：

RRF 公式：`score = 1 / (k + rank + 1)`，rank 从 0 开始。

- k=60 是业界对**大规模检索（百万文档）**的经验值，用于抑制 top-1 的过度主导
- 但当前知识库才 53 切片，`candidate_k = top_k * 3 = 15`，候选池很小
- 结果：**所有分数被压得很平**（实测 ~0.03），不同 rank 之间的区分度被严重稀释

**区分度对比**：

| k 值 | rank=0 分数 | rank=1 分数 | 区分度（差值） |
|------|------------|------------|--------------|
| 60（当前） | 0.0164 | 0.0161 | 0.0003 |
| 20 | 0.0476 | 0.0455 | 0.0021 |
| 10 | 0.0909 | 0.0833 | 0.0076 |

**建议方案**：

短期：直接把 `rrf_k` 降到 20。

进阶：**让 k 跟着候选规模自适应**：

```python
def _rrf_fusion(self, vector_results, keyword_results, k=None):
    k = k or settings.rrf_k
    total_candidates = len(vector_results) + len(keyword_results)
    if total_candidates < k // 2:
        k = max(10, total_candidates // 2)
    ...
```

**优先级**：🟡 P1
**工作量**：5 分钟（改配置）/ 20 分钟（自适应）
**收益**：显著提升分数区分度

### 问题 3.2 🔴 RRF 完全丢失原始分数信息（最深层设计问题）

**位置**：`rag/retriever.py:_rrf_fusion()`

**现状代码**：

```python
def _rrf_fusion(self, vector_results, keyword_results, k=None):
    k = k or settings.rrf_k
    scores = {}
    doc_map = {}
    
    for rank, doc in enumerate(vector_results):
        key = doc.get("chunk_id") or doc.get("content", "")[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)  # ← 只用 rank
        doc_map[key] = doc
    
    for rank, doc in enumerate(keyword_results):
        key = doc.get("chunk_id") or doc.get("content", "")[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)  # ← 只用 rank
        ...
```

**问题分析**：

RRF 只看 **rank（排名）**，完全不看原始 cosine 相似度或 BM25 分数。后果：

- 向量检索返回的 cos=0.95 和 cos=0.31 **只要都在 rank=0，对 RRF 的贡献一模一样**
- 一个 cos=0.95 的强匹配 vs cos=0.31 的弱匹配，得分差距只来自 rank 差，**严重失真**

**典型翻车场景**：用户问 "LangChain 的 LCEL 是什么"

| 检索器 | top1 | 原始分数 | RRF 贡献 |
|--------|------|---------|---------|
| 向量检索 | 真正讲 LCEL 的段落 | cos=0.92 | 1/(60+0+1)=0.0164 |
| BM25 | 含"LangChain"关键词但语义无关的段落 | bm25=15 | 1/(60+0+1)=0.0164 |

→ 真正的答案可能因为 rank 差被排到第 2。

**建议方案（三选一，复杂度递增）**：

**方案 A（最小改动）— 加权 RRF**：

给向量检索更高权重（语义匹配更可靠）：

```python
VECTOR_WEIGHT = 1.5
KEYWORD_WEIGHT = 0.8

for rank, doc in enumerate(vector_results):
    scores[key] = scores.get(key, 0) + VECTOR_WEIGHT / (k + rank + 1)
for rank, doc in enumerate(keyword_results):
    scores[key] = scores.get(key, 0) + KEYWORD_WEIGHT / (k + rank + 1)
```

**方案 B（推荐）— 加上分数归一化的凸组合（Convex Combination）**：

保留原始分数信息，归一化后加权：

```python
def _normalize_scores(results):
    if not results: return results
    max_s = max(r["score"] for r in results)
    min_s = min(r["score"] for r in results)
    rng = max_s - min_s or 1
    for r in results:
        r["norm_score"] = (r["score"] - min_s) / rng
    return results

vector_results = _normalize_scores(vector_results)
keyword_results = _normalize_scores(keyword_results)

ALPHA = 0.6  # 向量权重
for rank, doc in enumerate(vector_results):
    rrf_v = 1 / (k + rank + 1)
    scores[key] = scores.get(key, 0) + ALPHA * doc["norm_score"] + (1-ALPHA) * rrf_v
```

**方案 C（终极）— 引入 Reranker 模型**：

拉一个 `bge-reranker-v2-m3`，对 RRF 后的 top-20 做精排。工业级做法，准确率最高，但要多一次模型调用（~100ms）。

**优先级**：🟡 P1
**工作量**：方案 A 半小时 / 方案 B 1 小时 / 方案 C 半天
**收益**：检索质量核心提升

### 问题 3.3 🔴 RRF 阈值对 cosine/RRF 两种分数不加区分

**位置**：`rag/retriever.py:search()` + `config/settings.py:29`

**现状代码**：

```python
rrf_threshold: float = 0.015

def search(self, query, top_k=None):
    ...
    if keyword_results:
        fused_results = self._rrf_fusion(vector_results, keyword_results)
    else:
        fused_results = vector_results  # ← 走原始 cosine similarity！
    
    # 然后用同一个阈值过滤
    if rrf_threshold > 0:
        fused_results = [r for r in fused_results if r.get("score", 0) >= rrf_threshold]
```

**问题分析**：

这是**重大问题**。当 BM25 没结果时（查询很冷门），`fused_results` 直接是向量结果，**score 是 cosine 相似度（值域 [-1,1]，常见 0.3~0.9）**；当 BM25 有结果时，**score 是 RRF 分数（值域 0~0.05）**。

然后同一个 `0.015` 阈值过滤两种完全不同量级的分数：

| 场景 | score 语义 | 典型值域 | 0.015 阈值效果 |
|------|-----------|---------|--------------|
| 有 BM25 结果 | RRF 分数 | 0~0.05 | 几乎不过滤（全保留） |
| 无 BM25 结果 | cosine 相似度 | 0.3~0.9 | 不过滤任何东西 |

**这个阈值形同虚设**。

**建议方案**：分开阈值，分开处理：

```python
# settings.py
similarity_threshold: float = 0.3      # 向量相似度阈值（启用已存在的 dead config）
rrf_threshold: float = 0.005           # RRF 阈值，降低

# retriever.py search()
if keyword_results:
    fused = self._rrf_fusion(vector_results, keyword_results)
    fused = [r for r in fused if r["score"] >= settings.rrf_threshold]
else:
    # 纯向量路径用 similarity_threshold
    fused = [r for r in vector_results if r["score"] >= settings.similarity_threshold]
```

**优先级**：🔴 P1
**工作量**：20 分钟
**收益**：阈值真正生效
