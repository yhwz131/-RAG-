## 一、演进动机与核心判断

### 现状

当前项目（`graduation-project` 分支）是一个**内置向量库的单体 RAG**：

```
本地文件 → 切片 → embedding → Milvus Lite（本地）→ 检索 → 对话
                                          ↑
                                       数据"住"在项目里
```

数据源和检索逻辑耦合在一个进程内，Milvus Lite 就是项目的"内部存储"。

### 核心洞察

> "向量数据库也可以像 SQL 一样主要通过外部对接的方式来引用。该项目可以转变为主要做检索、对话等功能的上游 RAG 部分，而数据来源可以从任何有数据库的地方。"

这个洞察指向了正确的演进方向：**项目应该专注做"检索编排",数据来源可插拔。**

### 核心判断

**方向正确，但"向量库像 SQL 一样外部化"这句话里藏着一个关键技术区别必须先讲清——否则会设计出错。** SQL 外部化和向量库外部化是两个难度完全不同的事情，不能简单类比。

---

## 二、关键技术区别：SQL 外部化 vs 向量库外部化

### SQL 外部化：为什么容易

SQL 数据库外部化简单，是因为它的**协议和查询语言是标准的**：

- MySQL 协议、PostgreSQL 协议都是国际标准
- 查询语言是 SQL，统一
- 项目只需知道"怎么连、怎么查"，数据在本地还是远程无所谓
- 数据库换主机，项目只改一个连接字符串

**当前的 T2SQL 已经是这个模式**——`DatabaseSource` 抽象 + 外部连接，数据不在项目里。这是真正的低耦合。

### 向量库外部化：难点不在"存储"，在"质量链"

很多人误以为"把 Milvus Lite 换成外部 Milvus 集群"就是向量库外部化。**这只是存储位置变了，本质没变**——项目还是要管一整条质量链：

```
文档 → 切片 → embedding → 入库 → 索引 → 检索 → 重排
       ↑___________________________________________↑
       这整条链是 RAG 的核心价值，不是"存储"
```

向量检索的**质量**取决于四个因素，它们和数据强绑定，**不能简单外包给"外部向量库"**：

| 质量因素 | 谁负责 | 能否外部化 |
|---------|--------|-----------|
| 切片策略 | 项目 | ❌ 这是核心竞争力 |
| Embedding 模型选型 | 项目 | ❌ 决定检索上限 |
| 索引参数（HNSW M/ef） | 项目 | ⚠️ 可委托给向量库，但要懂 |
| 重排模型 | 项目 | ❌ 决定最终精度 |

**真正的"外部化"是：数据源（文档/SQL/API）在外部，但项目掌握"数据→向量→检索→对话"的整条质量链。**

### 两个层次的区分

| 层次 | 含义 | 难度 | 价值 |
|------|------|------|------|
| **浅层：存储外部化** | Milvus Lite → 外部 Milvus 集群 | 低 | 只是换个存储位置 |
| **深层：数据源抽象** | 文档/SQL/API 都是可插拔外部源，项目做检索编排 | 高 | 这才是真正的转变 |

**本设计的演进目标是深层——数据源抽象。** 存储是否外部化（浅层）是规模化的可选决策，不是演进的前提。

## 三、目标架构：三层分离

基于核心洞察，项目应演进为三层分离架构：

```
┌─────────────────────────────────────────────────────┐
│  对话编排层（Agent / 路由 / 记忆）                    │
│  - LangGraph Agent                                   │
│  - router 规则层                                     │
│  - ConversationMemory                               │
└──────────────────────┬──────────────────────────────┘
                       │ 调用检索能力
┌──────────────────────▼──────────────────────────────┐
│  检索质量链（核心竞争力，不外包）                      │
│  - 切片策略（RecursiveCharacterTextSplitter）        │
│  - Embedding（bge-large-zh）                         │
│  - 检索（向量 + BM25 + RRF / Text-to-SQL）           │
│  - Reranker（bge-reranker-v2-m3）                    │
└──────────────────────┬──────────────────────────────┘
                       │ 通过适配层访问数据
┌──────────────────────▼──────────────────────────────┐
│  数据源适配层（可插拔，外部对接）                      │
│  - VectorStoreAdapter（Milvus / Qdrant / PG-Vector） │
│  - DatabaseSource（MySQL / PostgreSQL）  ← 已有       │
│  - DocumentSource（本地文件 / OSS / API）            │
│  - APISource（REST / GraphQL）                       │
└─────────────────────────────────────────────────────┘
```

### 三层的职责边界

| 层 | 职责 | 不做什么 |
|----|------|---------|
| **对话编排层** | 决定调哪个数据源、如何组合结果、管理多轮对话 | 不关心数据从哪来 |
| **检索质量链** | 切片/embedding/检索/重排——决定检索质量 | 不关心数据存在哪 |
| **数据源适配层** | 连接、鉴权、读写、同步具体数据源 | 不做质量决策 |

### 关键设计原则

1. **检索质量链必须自己掌握**——这是 RAG 的核心价值，不能委托给外部服务
2. **数据源适配层全部可插拔**——换数据源只改 adapter，不动 RAG 主体
3. **对话编排层独立于数据源**——LangGraph Agent 只调统一接口，不关心背后是什么

## 四、数据源适配层设计

### 统一抽象

将现有的 `DatabaseSource` 思路扩展为统一的 `DataSource` 抽象：

```python
from abc import ABC, abstractmethod
from typing import List, Dict, Optional, Any

class DataSource(ABC):
    """统一数据源抽象——所有外部数据源实现这套接口"""

    @property
    @abstractmethod
    def source_type(self) -> str:
        """数据源类型标识"""
        pass

    @abstractmethod
    def connect(self) -> bool:
        """测试连接"""
        pass

    @abstractmethod
    def fetch(self, query: str, **kwargs) -> List[Dict]:
        """按查询获取数据

        不同数据源语义不同：
        - 向量库：做相似度检索
        - SQL：做 Text-to-SQL
        - 文档源：做关键词/全文检索
        - API：调用对应接口
        """
        pass

    def close(self):
        """关闭连接"""
        pass
```

### 各适配器职责

| 适配器 | 实现类 | 当前状态 | 外部化路径 |
|--------|--------|---------|-----------|
| **VectorStoreAdapter** | `MilvusLiteAdapter` → `MilvusClusterAdapter` / `QdrantAdapter` | 本地 Lite | 按 SDK 切换 |
| **DatabaseSource** | `MySQLSource` / `PostgreSQLSource` | ✅ 已有抽象 | 已外部化 |
| **DocumentSource** | `LocalFileAdapter` → `OSSAdapter` / `S3Adapter` | 本地文件 | 对接对象存储 |
| **APISource** | `RESTAdapter` / `GraphQLAdapter` | 无 | 对接外部系统 |

### VectorStoreAdapter 设计示例

```python
class VectorStoreAdapter(DataSource):
    """向量库适配器——可换 Milvus/Qdrant/PG-Vector"""

    def __init__(self, backend: str = "milvus_lite"):
        self.backend = backend
        # 根据 backend 选择具体实现
        ...

    @property
    def source_type(self) -> str:
        return f"vector_{self.backend}"

    def fetch(self, query: str, top_k: int = 5, **kwargs) -> List[Dict]:
        """向量相似度检索"""
        query_vector = self.embedder.embed_query(query)
        return self._ann_search(query_vector, top_k, **kwargs)

    def _ann_search(self, vector, top_k, **kwargs):
        """ANN 检索——具体实现委托给底层 SDK"""
        # Milvus / Qdrant / PG-Vector 各自实现
        ...

    def insert_documents(self, chunks: List[Dict]) -> int:
        """入库（向量库特有，不在 fetch 接口内）"""
        ...
```

**关键**：`insert_documents` 是向量库特有的能力（文档源/API 源没有），作为适配器的扩展方法存在，但核心 `fetch` 接口统一。

## 五、检索质量链的归属

### 哪些必须自己掌握

以下四步是 RAG 的核心价值，**哪怕数据源外部化，这些逻辑也必须在项目里**：

| 步骤 | 为什么不能外包 |
|------|--------------|
| **切片策略** | 决定召回上限，是 RAG 的核心竞争力 |
| **Embedding 模型选型** | 决定向量质量，直接影响检索精度 |
| **检索算法（混合检索/RRF）** | 召回质量的核心，决定"能否找到" |
| **重排模型（Reranker）** | 精排质量的核心，决定"排序对不对" |

### 哪些可以委托给外部向量库

外部向量库只承担两件事：

1. **存储向量数据**
2. **执行 ANN（近似最近邻）检索**

**质量相关的决策（切多细、用什么 embedding、怎么重排）全在项目里。**

### 边界示意

```
项目边界（质量链）          │  外部向量库边界（存储 + ANN）
                          │
切片 → embedding → ────────►│──→ 存储向量
                          │
检索请求 → ────────────────►│──→ ANN 检索返回 top-N
                          │
top-N → RRF → Reranker ────│  （项目内精排）
```

## 六、与现有架构的映射

当前 `graduation-project` 分支的代码，如何映射到三层架构：

| 三层架构 | 现有代码 | 演进动作 |
|---------|---------|---------|
| **对话编排层** | `rag/chain.py` + `rag/router.py` | LangGraph 重构（见 Agent 升级文档） |
| **检索质量链** | `rag/retriever.py` + `embeddings/*` | 加 Reranker、优化切片、修复 BM25 |
| **数据源适配层** | `api/pipeline/engines/database.py`（已有 DatabaseSource 抽象） | 抽象 VectorStoreAdapter、扩展 DocumentSource |

### 现有的低耦合设计（值得保留）

当前已经有三个低耦合设计，是演进的基石：

1. **`DatabaseSource` 抽象基类**（`adapter.py`）——RAG 不直接依赖 MySQL/PG 实现
2. **单一入口**（`chain.py:_handle_database_query`）——所有 DB 交互收口
3. **路由层隔离**（`router.py`）——database 是四条平行路径之一

**这三个设计证明项目已经在朝数据源抽象的方向走，只是还没有把向量库纳入同一个抽象。** 演进就是补全这一块。

## 七、演进路径（三步走）

### 核心原则

**不要一步到位做成"全外部化"。先做逻辑抽象，别急着物理拆分。** `DataSource` 接口让数据源可插拔，但项目还是一个进程——这是性价比最高的方案。

### 第一步：完成抽象（不换实现）

**目标**：建立数据源适配层，但 Milvus Lite 还在本地。

**动作**：

1. 定义 `DataSource` / `VectorStoreAdapter` 统一抽象
2. 把现有 Milvus 操作包装成 `MilvusLiteAdapter`
3. 把现有 `DatabaseSource` 并入统一抽象（已有基础，微调）
4. **检索质量链完全不动**

**价值**：架构清晰了，但运行时行为不变，零风险。

**验收**：
- [ ] `DataSource` 抽象定义完成
- [ ] Milvus 操作全部走 `MilvusLiteAdapter`
- [ ] `DatabaseSource` 并入统一抽象
- [ ] 现有功能全部正常（四条路由链路不变）

### 第二步：选一个数据源真外部化

**目标**：验证外部数据源能跑通。

**推荐先外部化文档源**（`DocumentSource`）——把"本地文件上传"改成"对接 OSS/S3/企业网盘"。理由：

- 文档源的协议相对简单（就是拉文件）
- 不涉及复杂的检索语义
- 验证外部化的工程链路（鉴权、同步、增量）

**动作**：

1. 实现 `OSSAdapter` / `S3Adapter`
2. 上传接口增加"从 URL 拉取"模式
3. 增量同步机制（只处理新增/变更的文档）

**验收**：
- [ ] 能从外部对象存储拉取文档入库
- [ ] 鉴权机制有效
- [ ] 增量同步能识别变更

### 第三步：向量库按需外部化

**目标**：仅在规模化需求出现时再做。

**触发条件**（满足任一即可考虑）：

- 单机 Milvus Lite 扛不住（数据量 > 100万向量）
- 需要多服务共享同一向量库
- 需要向量库高可用
- 需要对接客户的现有向量库

**动作**：

1. 实现 `MilvusClusterAdapter`（外部 Milvus 集群）
2. 迁移存储后端
3. 网络层优化（连接池、重试）

**关键提醒**：**在这之前，Milvus Lite 留在本地完全没问题。** Lite 到几百万向量都没问题，企业知识库场景大概率用不到集群版。

## 八、收益与代价

### 收益（很大）

| 收益 | 说明 |
|------|------|
| **真正的多源 RAG** | 一个查询可以同时查向量库 + SQL + 外部 API，Agent 编排 |
| **数据源可替换** | 换向量库只改 adapter，不动 RAG 主体 |
| **职责清晰** | 项目专注"检索编排"，不背"数据存储"的包袱 |
| **契合 Agent 平台定位** | LangGraph Agent 天然适合"调用多个数据源 tool"的模式 |
| **可商业化** | 这是 RAG as a Service 的架构，能对接任意客户的数据 |

### 代价（必须正视）

| 代价 | 说明 |
|------|------|
| **复杂度上升** | 多一层抽象，调试链路变长 |
| **质量链不能省** | 别以为"数据外部化"就能少做事，切片/embedding/重排还是你的 |
| **运维边界模糊** | 向量库挂了算谁的？数据不一致怎么排查？ |
| **延迟增加** | 跨网络调用外部数据源，比本地 Lite 慢 |
| **一致性挑战** | 文档更新后，外部向量库的索引何时刷新？ |

**最大的代价是第三条**——当数据源从"项目内部"变成"外部服务"，**一致性和可用性的责任边界就模糊了**。这是分布式系统的经典难题，单体应用不用操心。

## 九、关键设计原则

### 原则 1：检索质量链不能外包

切片、embedding、检索、重排——这四步是 RAG 的核心价值，不能委托给外部服务。

哪怕向量库换成外部的，这四步的逻辑还是在项目里。外部向量库只承担存储 + ANN 检索，不承担质量决策。

### 厧则 2：先做逻辑抽象，别急着物理拆分

`DataSource` 接口让数据源可插拔，但项目还是一个进程。

**真正的外部化是逻辑层面的（抽象边界清晰），不是物理层面的（拆成多个服务）。** 等真有规模化需求，再拆不迟。

### 原则 3：抵制"微服务"诱惑

演进过程中很容易演变成"把 RAG 拆成微服务"。**强烈建议抵抗这个诱惑**：

1. 单人/小团队项目，微服务的运维成本会吃掉所有开发精力
2. RAG 的核心价值在质量链，不在架构
3. 单体 + 良好抽象 = 最优解

### 原则 4：Milvus Lite 可以一直留本地

存储外部化是规模化需求出现后的可选决策，不是演进的前提。

**除非出现以下信号，否则不要外部化存储**：
- 数据量超过 Milvus Lite 的舒适区
- 多服务共享向量库需求
- 高可用要求

### 原则 5：保留现有的路由规则层

当前 `router.py` 的四分类路由（chitchat/general/rag/database）是零成本拦截简单流量的有效机制。

**演进后仍应保留路由层作为 Agent 的前置网关**——简单查询直接走快速路径，只有复杂/多步查询才进 Agent。这个价值在任何架构下都不该丢。

4. **第四阶段（外部化验证，2 周）**：文档源外部化（OSS/S3）
5. **第五阶段（按需）**：向量库外部化、全链路 Agent 化

**每个阶段独立可验证、可回滚。** 不要跳步，不要合并阶段。

## 一页纸总结

**核心命题**：项目从"内置向量库的单体 RAG"演进为"数据源可插拔的检索编排平台"。

**关键认知**：

1. 向量库外部化 ≠ SQL 外部化——难点不在"存储"在"质量链"
2. 真正的外部化是"逻辑层面解耦"，不是"物理层面拆服务"
3. 检索质量链（切片/embedding/检索/重排）是核心价值，不能外包
4. 数据源适配层（VectorStore/Database/Document/API）全部可插拔

**三步走路径**：

1. **先完成抽象**（不换实现，零风险）
2. **再外部化文档源**（验证工程链路）
3. **最后按需外部化向量库**（规模化触发）

**最大风险**：不是技术选型，是"一次性大重构把能跑的系统搞挂"。渐进迁移是唯一稳妥的路径。

**最大诱惑**：把项目拆成微服务。单体 + 良好抽象 = 单人/小团队的最优解。
