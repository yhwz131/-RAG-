## 1. 背景与动机

### 1.1 开题报告与当前项目的差距

开题报告题目为 **"面向企业大数据的智能知识库问答系统的设计与开发"**，核心关键词是"大数据"。但当前项目实现的是一个纯 RAG 文档问答系统，虽然 RAG 链路完整，但在"大数据"维度上存在明显缺失：

| 开题报告要求 | 当前实现状态 | 差距评估 |
|-------------|------------|---------|
| PySpark 数据处理 | `spark/processor.py` 已有框架代码 | ⚠️ 仅骨架，缺少清洗/去重/统计逻辑 |
| 多源数据接入 | 仅支持文件上传 | ⚠️ 缺少数据库、API 等数据源 |
| 数据清洗/去重/标准化 | 无 | ❌ 完全缺失 |
| 数据质量监控 | 无 | ❌ 完全缺失 |
| 分析报表生成 | 无 | ❌ 完全缺失 |
| RAG 问答链路 | 完整 | ✅ 已实现 |
| 向量检索 + BM25 混合检索 | 完整 | ✅ 已实现 |
| 多模态支持 | 完整 | ✅ 已实现 |

### 1.2 核心矛盾

- **专业要求**：大数据技术专业，答辩必须体现大数据处理能力
- **项目本质**：RAG 知识问答系统，核心价值在检索与生成质量
- **解决思路**：将大数据处理作为现有系统的**可选处理引擎**，而非独立的上游系统

### 1.3 设计思路

| 方案 | 描述 | 评价 |
|------|------|------|
| A. 独立上游管线 | 完整的 PySpark ETL + 独立前端 | ❌ 过度工程，与核心功能脱节 |
| B. 仅 CLI 脚本 | 命令行跑 Spark，无前端交互 | ⚠️ 答辩时缺乏可视化，用户不友好 |
| **C. 前端统一入口 + 用户选择模式** | **前端上传时选择处理模式，一站式完成** | ✅ **用户友好，答辩直观** |

最终方案的核心思路：**前端是唯一的用户入口**，上传页面提供"快速入库"和"大数据处理"两种模式供用户选择，后端根据选择走不同的处理链路。

## 2. 现状分析

### 2.1 当前数据处理流程

```
当前流程（直接入库）：

用户上传文件
    │
    ▼
POST /api/docs/upload
    │
    ├── FileParser.parse_with_pages()     ← 解析文件为文本
    ├── TextChunker.chunk_with_pages()    ← 切片
    ├── VectorRetriever.insert_documents() ← 向量化写入 Milvus
    ├── MultimodalRetriever (可选双写)     ← 多模态 Collection
    └── 返回 { filename, chunks, message }
```

**特点**：
- 同步处理，上传即入库
- 无数据清洗环节（不做去重、格式标准化）
- 无处理统计和质量报告
- 适合单文件、小批量场景

### 2.2 现有 Spark 模块分析

`spark/processor.py` 已有两种处理模式：

| 模式 | 函数 | 说明 |
|------|------|------|
| `local` | `process_local()` | 纯 Python 本地处理，不依赖 Spark |
| `spark` | `process_spark()` | PySpark 并行处理，RDD parallelize |

**已有功能**：文件扫描、解析、切片、输出 JSON
**缺失功能**：数据清洗、去重、统计分析、质量报告、自动导入 Milvus

## 3. 改进目标

### 3.1 核心目标

将现有的单一上传入口升级为**双模式处理**，用户在前端选择处理模式，后端自动走对应链路：

```
统一入口（前端上传页面）：
    用户选择模式 → 上传文件 → 后端自动处理 → 入库

    ┌─ 快速入库模式：上传 → 解析 → 切片 → 入库（同步，秒级）
    └─ 大数据模式：  上传 → Spark 处理 → 清洗 → 统计 → 切片 → 入库（异步，完成后通知）
```

### 3.2 具体改进项

| 编号 | 改进项 | 优先级 | 说明 |
|------|--------|--------|------|
| 1 | 增强 Spark 处理管线 | P0 | 加入清洗、去重、统计逻辑 |
| 2 | 后端批量处理 API | P0 | 接收前端上传 → 触发 Spark 处理 → 自动入库 |
| 3 | 后端管线状态 API | P0 | 返回处理统计、导入历史 |
| 4 | 前端上传页面改造 | P0 | 加入处理模式选择（快速入库 / 大数据处理） |
| 5 | 前端管线状态页面 | P1 | 只读展示处理结果和统计 |
| 6 | start.sh 集成 | P1 | `./start.sh pipeline` 命令（CLI 备选入口） |

### 4.2 两种处理模式对比

用户在前端上传页面选择处理模式，后端自动走对应链路：

| 适用场景 | 日常少量上传、临时补充 | 首次批量导入、定期数据更新 |
| 结果展示 | 上传成功提示（切片数） | 上传成功 + `/admin` 页面完整统计 |

### 5.4 Spark 处理伪代码（process_spark 增强版）

```python
def process_spark_enhanced(input_dir: str, output_dir: str):
    """增强版 Spark 处理：解析 → 清洗 → 切片 → 统计 → 输出"""
    spark = SparkSession.builder \
        .appName(settings.spark_app_name) \
        .master(settings.spark_master) \
        .getOrCreate()

    # 1. 文件扫描
    files = scan_files(input_dir)
    rdd = spark.sparkContext.parallelize(files, numSlices=min(len(files), 8))

    # 2. 并行解析
    parsed_rdd = rdd.map(parse_file).filter(lambda x: x is not None)

    # 3. 数据清洗（去重、去空、编码标准化）
    cleaned_rdd = parsed_rdd \
        .filter(lambda x: len(x["text"].strip()) >= 50) \
        .map(lambda x: (hash(x["text"]), x)) \
        .reduceByKey(lambda a, b: a) \
        .map(lambda x: x[1])

    # 4. 切片
    chunks_rdd = cleaned_rdd.flatMap(lambda doc: chunk_document(doc))

    # 5. 收集结果
    all_chunks = chunks_rdd.collect()

    # 6. 统计分析（单机执行，数据量已缩小）
    stats = analyze_statistics(all_chunks, clean_stats)
    quality = analyze_quality(all_chunks, failed_files)

    # 7. 输出
    export_results(all_chunks, stats, quality, output_dir)
    spark.stop()
```

@router.post("/process")
async def process_batch_docs(
    background_tasks: BackgroundTasks,
    files: List[UploadFile] = File(...)
):
    """
    大数据处理模式入口：
    1. 将上传文件保存到 data/raw/
    2. 后台异步触发处理管线（通过 PipelineAdapter）
    3. 处理完成后自动导入 Milvus

    saved_files = []
    for file in files:
        file_path = Path(settings.upload_dir).parent / "raw" / file.filename
        file_path.write_bytes(await file.read())
        saved_files.append(str(file_path))

    # 2. 生成任务 ID，标记为"处理中"
    task_id = str(uuid.uuid4())[:8]
    save_task_status(task_id, "processing", files=len(saved_files))

    # 3. 后台异步执行处理管线（通过适配器，不直接依赖 Spark）
    background_tasks.add_task(
        run_pipeline_background, task_id, saved_files
    )

    return {
        "task_id": task_id,
        "status": "processing",
        "files_count": len(saved_files),
        "message": "大数据处理已启动，请到「管线状态」页面查看进度"
    }


async def run_pipeline_background(task_id: str, files: List[str]):
    """后台执行：处理管线 → 自动导入 Milvus"""
    try:
        # 通过适配器调用，不直接依赖 Spark 实现
        result = await asyncio.to_thread(
            pipeline_adapter.run,
            input_dir=settings.raw_dir,
            output_dir=settings.processed_dir
        )

        # 自动导入 Milvus（JSON 契约，解耦）
        chunks_file = Path(settings.processed_dir) / "all_chunks.json"
        chunks = json.loads(chunks_file.read_text(encoding="utf-8"))
        count = retriever.insert_documents(chunks)

        save_task_status(task_id, "completed",
                        chunks=count, stats=result["stats"])
    except Exception as e:
        save_task_status(task_id, "failed", error=str(e))

### 6.3 处理管线适配器（解耦关键）

**这是保持低耦合的核心设计。** 后端 API 不直接调用 Spark，而是通过适配器接口调用：

```python
# api/pipeline_adapter.py（新建）
"""
处理管线适配器 — 解耦后端 API 与具体处理引擎

后端只依赖这个接口，不关心底层是 Spark、本地处理还是其他引擎。
切换引擎只需改这一处，不影响任何其他代码。
"""

from abc import ABC, abstractmethod
from typing import Dict, Any


class PipelineEngine(ABC):
    """处理引擎抽象接口"""

    @abstractmethod
    def run(self, input_dir: str, output_dir: str) -> Dict[str, Any]:
        """
        执行处理管线。
        输入：原始文件目录
        输出：处理结果（JSON）到 output_dir
        返回：统计信息
        """
        pass


class SparkPipelineEngine(PipelineEngine):
    """PySpark 处理引擎"""

    def run(self, input_dir: str, output_dir: str) -> Dict[str, Any]:
        from spark.processor import process_local
        stats = process_local(input_dir, output_dir)
        return {"engine": "spark", "stats": stats}


class SimplePipelineEngine(PipelineEngine):
    """简单处理引擎（无 Spark 依赖，单机处理）"""

    def run(self, input_dir: str, output_dir: str) -> Dict[str, Any]:
        from utils.file_parser import FileParser
        from embeddings.chunker import TextChunker
        # 单机版本：解析 → 清洗 → 切片 → 输出
        # 功能等价，但不依赖 Spark
        ...
        return {"engine": "simple", "stats": stats}


# 全局实例 — 切换引擎只需改这一行
# 或者从 settings 中读取配置动态选择
pipeline_adapter: PipelineEngine = SparkPipelineEngine()
```

**为什么这样设计？**

```
后端 API (routes_docs.py)
    │
    │ 只依赖抽象接口 PipelineEngine.run()
    │ 不 import spark.processor
    │
    ▼
pipeline_adapter (抽象层)
    │
    ├── SparkPipelineEngine   ← 生产环境，PySpark 并行处理
    ├── SimplePipelineEngine  ← 测试/轻量场景，无 Spark 依赖
    └── 未来可扩展：Airflow、Dask 等
    │
    ▼
输出标准化 JSON → data/processed/all_chunks.json
    │
    ▼
retriever.insert_documents()  ← RAG 链，只消费 JSON，零耦合
```

后端 API 与 Spark 之间多了一层适配器，代码上没有直接 import 关系。即使 Spark 没装，后端照样跑（降级到 SimplePipelineEngine）。**这就是低耦合。**

## 10. 答辩要点

### 10.1 论文包装话术

> "系统采用**双模式统一入口架构**：前端上传页面提供「快速入库」和「大数据处理」两种模式供用户选择。快速入库模式适用于日常少量文档，即时完成解析、切片和向量化入库；大数据处理模式基于 PySpark 分布式处理管线，支持数据清洗、去重、质量分析等企业级数据处理能力，处理完成后自动入库。两种模式共享同一套向量化和检索链路，通过标准化数据契约实现解耦，用户无需了解底层技术差异，通过前端界面即可灵活选择。"

### 10.2 答辩可能的问题与回答

| 问题 | 回答要点 |
|------|---------|
| 为什么用 Spark？ | 企业级文档量大，单机处理有瓶颈；Spark 支持分布式扩展和 HDFS/S3 数据源 |
| 两种模式有什么区别？ | 快速入库是同步即时处理，适合单文件；大数据处理是批量异步处理，支持清洗去重和统计 |
| 用户怎么选择模式？ | 前端上传页面提供两种模式选项，附有清晰的使用场景描述，用户根据文档数量和需求选择 |
| 数据清洗做了什么？ | 去空文档、去重复（MD5 哈希）、编码标准化（UTF-8）、过短文档过滤 |
| 统计分析有什么用？ | 数据质量监控、知识库健康度评估、为后续检索优化提供数据支撑 |
| 为什么不自动判断模式？ | 用户最了解自己的需求，手动选择更可控、更透明，避免自动判断的误判风险 |

### 10.3 体现大数据能力的关键点

1. **PySpark 并行处理**：RDD parallelize + map/filter/reduceByKey
2. **数据清洗 Pipeline**：ETL 标准流程（Extract → Transform → Load）
3. **统计分析**：词频统计、数据质量报告、格式分布分析
4. **双引擎架构**：单机模式 + 分布式模式自适应
5. **标准化数据契约**：JSON Schema 解耦处理与入库
