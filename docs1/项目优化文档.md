# 知识问答系统 — 项目优化文档

> 版本：v1.0 | 日期：2026-06-22 | 基于当前代码库分析

---

## 一、现状分析

### 1.1 当前架构概览

系统采用**双链路架构**，同时维护纯文本和多模态两套完整链路：

```
用户请求 → routes_chat.py (mode="text"/"multimodal")
    ├─ 纯文本链路
    │   ├─ EmbeddingClient (bge-large-zh-v1.5, 1024 维)
    │   ├─ VectorRetriever (collection: knowledge_base)
    │   └─ RAGChain (is_multimodal=False, LLM: mimo-v2.5)
    └─ 多模态链路
        ├─ MultimodalEmbedder (Qwen3-VL-Embedding-8B, 4096 维)
        ├─ MultimodalRetriever (collection: knowledge_base_mm)
        └─ RAGChain (is_multimodal=True, LLM: mimo-v2.5)
```

### 1.2 核心问题

| 问题 | 影响 |
|------|------|
| 双链路维护成本高 | 同一逻辑（检索、入库、删除）在两个 Retriever 中各写一遍 |
| 前端 mode 切换增加用户认知负担 | 用户需手动选择"纯文本/多模态"模式 |
| 纯文本 Embedding (bge-large-zh) 功能冗余 | 多模态 Embedding (Qwen3-VL) 完全可以处理纯文本 |
| 路由层与检索层耦合不够紧密 | 路由结果未传递给检索器做针对性优化 |
| BM25 每次查询重新构建索引 | 大规模文档场景下性能瓶颈 |

---

## 二、优化方案总览

```
优化优先级：
P0（必做）：删除纯文本模式，统一为多模态单链路
P1（重要）：查询路由优化 + 检索逻辑优化
P2（锦上添花）：Prompt 精简 + 性能优化
```

## 四、P1-A：查询路由优化

### 4.1 现状分析

当前路由采用四层策略（`rag/router.py`）：

```
第1层：规则匹配（零成本）
  ├─ GREETING_PATTERNS → chitchat
  └─ SMALL_TALK_PATTERNS → chitchat

第2层：数据库查询检测（零成本）
  └─ DB_QUERY_KEYWORDS ≥ 2 个命中 → database

第3层：通用常识启发式（零成本）
  └─ GENERAL_PATTERNS 匹配 → general

第4层：LLM 分类（有成本，~0.5s）
  └─ 发送 prompt 到 mimo-v2.5 → rag/chitchat/general/database
```

### 4.2 问题诊断

| 问题 | 表现 |
|------|------|
| 数据库关键词过于宽泛 | "查询"、"列表"、"多少"等词在非数据库场景也会出现 |
| 强模式覆盖不全 | "弹幕最多"、"播放量前10"等自然表达需要更强的模式匹配 |
| LLM 分类成本浪费 | 简单问题（如"谢谢"）已经命中规则但仍在第1层返回，不会走到LLM |
| 路由结果未传递给检索层 | 路由只决定"走哪条路"，但没有为检索层提供额外提示（如关键词权重） |
| 缺乏反馈机制 | 路由判断错误时无法自动修正 |

### 4.3 优化方案

#### 4.3.1 增强规则层（第1、2层）

## 五、P1-B：检索逻辑优化

### 5.1 现状分析

当前检索流程（`rag/retriever.py`）：

```
用户查询 → preprocess_query()
    → 向量检索 (MultimodalRetriever._vector_search) — COSINE, top_k*3
    → BM25 检索 (MultimodalRetriever._bm25_search)  — jieba 分词, top_k*3
    → RRF 融合 (_rrf_fusion)                         — k=60
    → 阈值过滤 (rrf_threshold=0.015)
    → 返回 top_k=5 条结果
```

### 5.2 问题诊断

| 问题 | 影响 |
|------|------|
| BM25 每次查询重新构建索引 | `_bm25_search()` 每次调用都 `jieba.cut` 全部文档 + 构建 BM25Okapi 对象 |
| 固定 top_k，不区分查询复杂度 | 简单问题和复杂问题都返回 5 条，可能过多或过少 |
| RRF 融合权重固定 | 向量检索和 BM25 检索各占 50%，未根据查询类型动态调整 |
| 向量检索使用 COSINE metric | 对于短查询 vs 长文档切片，COSINE 可能偏向长文本 |
| BM25 分词粒度单一 | jieba 默认分词对专业术语（如"RAG""Milvus"）处理不佳 |
| 缺乏查询意图感知的检索 | 不同查询类型（事实型 vs 分析型）应采用不同检索策略 |

### 5.3 优化方案

#### 5.3.1 BM25 索引缓存优化

**问题：** 每次 `_bm25_search()` 都重新分词 + 构建索引

**方案：** 在 `_update_bm25_cache()` 中预计算分词结果并缓存

```python
def _update_bm25_cache(self):
    """更新 BM25 检索缓存（预计算分词）"""
    # ... 加载文档到 self._bm25_docs ...
    
    # 预计算分词结果（新增）
    self._bm25_corpus_tokens = [
        list(jieba.cut(doc.get("content", ""))) 
        for doc in self._bm25_docs
    ]
    # 预构建 BM25 索引（新增）
    if self._bm25_corpus_tokens:
        from rank_bm25 import BM25Okapi
        self._bm25_index = BM25Okapi(self._bm25_corpus_tokens)
    else:
        self._bm25_index = None

def _bm25_search(self, query, top_k=5):
    """BM25 检索（使用缓存索引）"""
    if not self._bm25_index:
        return []
    
    query_tokens = list(jieba.cut(query))
    scores = self._bm25_index.get_scores(query_tokens)
    # ... 后续不变 ...
```

**收益：** 文档入库后只需构建一次索引，查询时直接使用缓存，BM25 检索速度提升 10-100 倍。

#### 5.3.2 动态 Top-K 调整

**根据查询复杂度调整检索数量：**

```python
def _estimate_query_complexity(query: str) -> str:

def search(self, query, top_k=None, route_hints=None):

    elif complexity == "complex":

        candidate_k = top_k * 3
    else:
        top_k = top_k or settings.top_k
        candidate_k = top_k * 3
    ...
```

#### 5.3.3 动态 RRF 权重

**根据查询类型调整向量检索和 BM25 的权重：**

```python
def _rrf_fusion(self, vector_results, keyword_results, k=None, 
                vector_weight=1.0, keyword_weight=1.0):
    """加权 RRF 融合"""
    k = k or settings.rrf_k
    scores = {}
    doc_map = {}
    
    for rank, doc in enumerate(vector_results):
        key = doc.get("chunk_id") or doc.get("content", "")[:100]
        scores[key] = scores.get(key, 0) + vector_weight / (k + rank + 1)
        doc_map[key] = doc
    
    for rank, doc in enumerate(keyword_results):
        key = doc.get("chunk_id") or doc.get("content", "")[:100]
        scores[key] = scores.get(key, 0) + keyword_weight / (k + rank + 1)
        if key not in doc_map:
            doc_map[key] = doc
    ...

# 调用示例：数据库查询偏向 BM25，知识库查询偏向向量
if query_type == QueryType.DATABASE:
    fused = self._rrf_fusion(vector_results, keyword_results, 
                             vector_weight=0.6, keyword_weight=1.4)
else:
    fused = self._rrf_fusion(vector_results, keyword_results, 
                             vector_weight=1.2, keyword_weight=0.8)
```

#### 5.3.5 查询意图感知检索

**根据查询意图采用不同检索策略：**

```python
def _classify_query_intent(query: str) -> str:
    """识别查询意图（事实型/分析型/导航型）"""
    # 事实型：寻找具体信息
    factual_patterns = [r'(是什么|什么是|谁是|在哪|什么时候|多少)']
    # 分析型：需要综合分析
    analysis_patterns = [r'(对比|比较|分析|总结|优缺点|区别|异同)']
    # 导航型：寻找特定文档/章节
    navigation_patterns = [r'(哪个文档|哪一章|哪一页|找一下|查看)']
    
    for p in factual_patterns:
        if re.search(p, query): return "factual"
    for p in analysis_patterns:
        if re.search(p, query): return "analytical"
    for p in navigation_patterns:
        if re.search(p, query): return "navigational"
    return "general"
```

**不同意图的检索策略：**

| 意图类型 | Top-K | 候选池 | RRF 权重 | 上下文策略 |

| analytical | 8 | 4× | 向量 1.2 / BM25 0.8 | 广覆盖，多而全 |
| navigational | 3 | 2× | 向量 1.0 / BM25 1.5 | 关键词精确匹配 |
| general | 5 | 3× | 向量 1.0 / BM25 1.0 | 默认策略 |

### 6.2 上下文压缩优化

**当前问题：** `_build_context()` 使用简单的 token 估算和截断

**方案：** 引入上下文压缩策略

```python
def _compress_context(self, docs: List[Dict], max_tokens: int) -> str:
    """智能上下文压缩"""
    # 1. 按相关度排序（已由 RRF 完成）
    # 2. 优先保留高相关度文档的完整内容
    # 3. 对低相关度文档只保留摘要/首段
    # 4. 如果仍然超限，移除最低相关度的文档
    
    parts = []
    current_tokens = 0
    for i, doc in enumerate(docs):
        content = doc["content"]
        doc_tokens = estimate_tokens(content)
        
        if current_tokens + doc_tokens <= max_tokens:
            parts.append(content)
            current_tokens += doc_tokens
        elif i < 2:  # 前2条高相关文档：截断保留
            remaining = max_tokens - current_tokens
            truncated = content[:int(remaining * 0.8)] + "..."
            parts.append(truncated)
            break
        else:  # 低相关文档：跳过
            break
    
    return "\n\n---\n\n".join(parts)
```

### 6.3 LLM 调用优化

**当前问题：** 每次查询都调用 LLM（即使闲聊/通用问题）

**方案：** 缓存常见问题的回答

```python
# 在 RAGChain 中添加简单缓存
class RAGChain:
    def __init__(self, ...):
        self._answer_cache = {}  # {query_hash: (answer, timestamp)}
        self._cache_ttl = 300    # 5 分钟缓存
    
    def chat(self, query, session_id, stream=False, ...):
        # 闲聊/通用问题可以缓存（不涉及知识库）
        if query_type in (QueryType.CHITCHAT, QueryType.GENERAL):
            cache_key = hash(query)
            if cache_key in self._answer_cache:
                answer, ts = self._answer_cache[cache_key]
                if time.time() - ts < self._cache_ttl:
                    return answer, [], query_type
        ...
```
