### 1.3 核心依赖

| 组件 | 用途 | 说明 |
|------|------|------|
| **FastAPI** | Web 框架 | API 服务层 |
| **Milvus Lite** | 向量数据库 | `pymilvus` 的 `MilvusClient` API |
| **bge-large-zh** | 文本 Embedding | 1024 维，纯文本链路 |
| **Qwen3-VL-Embedding-8B** | 多模态 Embedding | 4096 维，图片+文本链路 |

### 1.4 双链路架构

```
纯文本链路: bge-large-zh (1024d) → knowledge_base
多模态链路: Qwen3-VL-Embedding (4096d) → knowledge_base_mm
```

两条链路**独立存储、独立检索、合并返回**，互不影响。多模态链路失败不影响纯文本链路。

## 2. 开发工作流

### 2.1 标准开发流程

任何优化或功能开发，**必须严格遵循以下五步流程**：

```
步骤1: 阅读需求文档
  │
  ▼
步骤2: 分析问题、验证是否真实存在
  │
  ▼
步骤3: 实现修改（按优先级 P0 → P1 → P2）
  │
  ▼
步骤4: 测试验证（单元测试 → 完整链路测试）
  │
  ▼
步骤5: 同步更新 /docs1 下的相关文档
```

### 2.2 优先级定义

| 优先级 | 含义 | 处理要求 |
|--------|------|----------|
| **P0** | 必须修复，影响正确性或核心功能 | 立即修复，阻塞发布 |
| **P1** | 重要优化，影响用户体验或可维护性 | 尽快实现 |
| **P2** | 锦上添花，体验打磨 | 持续迭代 |

### 2.3 需求来源

开发需求统一记录在 `/docs1/` 目录下的文档中：

| 文档 | 用途 |
|------|------|
| `RAG优化建议文档.md` | 纯文本链路的优化方案与进度 |
| `多模态优化建议文档.md` | 多模态链路的优化方案与进度 |
| `RAG项目架构文档.md` | 整体架构设计，需与代码保持同步 |
| `1.txt` | 开发约束与基本规则 |

### 2.4 任务拆解原则

- 每个优化项独立可测，不耦合其他未完成的改动
- 大任务拆分为多个小步骤，每步完成后再进入下一步
- 使用 Todo List 跟踪所有任务进度，确保不遗漏

## 3. 测试规范

### 3.1 测试分层

```
┌─────────────────────────────────────┐
│  完整链路测试（端到端）              │  ← 步骤 4b
│  curl 请求 API，验证全流程           │
├─────────────────────────────────────┤
│  最小单元测试（函数级）              │  ← 步骤 4a
│  验证改动的每个函数/模块             │
└─────────────────────────────────────┘
```

### 3.2 最小单元测试规则

**对改动或新增的每个函数，必须逐一进行最小单元测试。**

```python
# 测试文件放在项目根目录，命名: test_<模块>.py
# 例如: test_multimodal.py, test_retriever.py

# 测试模式: 直接实例化目标类，调用目标函数，断言结果
def test_xxx():
    obj = SomeClass()
    result = obj.some_method(input)
    assert result == expected
    print("✅ 测试名称 通过")
```

测试要点：
- **覆盖每个改动函数**，不遗漏
- **Mock 外部依赖**：Milvus、LLM API、Embedding API 等用 `unittest.mock.patch` 模拟
- **测试边界条件**：空输入、异常输入、超长输入
- **测试降级路径**：API 超时、返回异常时应有 fallback 行为
- 全部测试通过后才能进入完整链路测试

### 3.3 完整链路测试规则

**单元测试全部通过后，必须进行完整链路测试，验证端到端可用性。**

```bash
# 1. 健康检查
curl -s http://127.0.0.1:8000/api/health | python3 -m json.tool

# 2. 纯文本模式（非流式）
curl -s -X POST http://127.0.0.1:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "测试问题", "session_id": "test-001", "stream": false, "mode": "text"}'

# 3. 多模态模式
curl -s -X POST http://127.0.0.1:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "测试问题", "session_id": "test-002", "stream": false, "mode": "multimodal"}'

# 4. 流式模式
curl -s -N -X POST http://127.0.0.1:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "测试问题", "session_id": "test-003", "stream": true, "mode": "multimodal"}'

# 5. 闲聊路由（应返回 chitchat 类型）
curl -s -X POST http://127.0.0.1:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "你好", "session_id": "test-004", "stream": false}'

# 6. 前端构建
cd frontend && npm run build && cd ..
```

链路测试要点：
- 验证 **所有模式**（text / multimodal / 流式 / 非流式）
- 验证 **查询路由**（rag / chitchat / general 三种类型）
- 验证 **返回字段完整性**（answer、references、query_type、session_id）
- 测试通过后才能视为开发完成

## 4. 代码规范

### 4.1 Python 代码风格

| 规则 | 说明 |
|------|------|
| **命名** | 模块/函数 `snake_case`，类 `PascalCase`，常量 `UPPER_CASE` |
| **类型注解** | 公共函数必须有参数和返回值类型注解 |
| **文档字符串** | 公共类和函数必须有 docstring（中文） |
| **日志** | 使用 `utils.logger.get_logger()` 获取 logger，不用 `print` |
| **导入顺序** | 标准库 → 第三方库 → 项目内部模块，用空行分隔 |

```python
# ✅ 正确示例
import hashlib
from typing import List, Dict, Optional

import httpx

from config.settings import settings
from utils.logger import get_logger

logger = get_logger("module_name")
```

### 4.2 HTTP 调用规范

本项目所有外部 HTTP 调用使用**同步 `httpx.Client`**，**不使用 `async/await`**。

```python
# ✅ 正确: 同步 httpx.Client
with httpx.Client(timeout=30) as client:
    resp = client.post(url, json=payload, headers=headers)
    resp.raise_for_status()
    return resp.json()

# ❌ 错误: 不要使用 async httpx.AsyncClient
# ❌ 错误: 不要使用 requests 库（项目统一用 httpx）
```

### 4.3 Milvus 操作规范

使用 `pymilvus.MilvusClient` API（非旧版 ORM API）：

```python
from pymilvus import MilvusClient

client = MilvusClient("./data/milvus.db")

# 查询
results = client.search(collection_name="knowledge_base", data=[query_vec], limit=5, output_fields=["content", "source"])

# 插入（带去重检查）
existing = client.query(collection_name="knowledge_base", filter=f'chunk_id == "{chunk_id}"', output_fields=["chunk_id"])
if not existing:
    client.insert(collection_name="knowledge_base", data=[record])
```

### 4.4 COSINE 相似度计算

Milvus COSINE 度量返回的是**距离（distance）**，需要转换为**相似度（similarity）**：

```python
# distance = 1 - cosine_similarity
# similarity = 1.0 - distance
similarity = 1.0 - distance
```

**切勿混淆 distance 和 similarity 的语义方向。**

### 4.5 Prompt 模板管理

所有 Prompt 模板集中在 `rag/prompt_template.py` 中定义：

| 模板 | 用途 |
|------|------|
| `SYSTEM_PROMPT` | 纯文本 RAG 模式 |
| `MULTIMODAL_SYSTEM_PROMPT` | 多模态 RAG 模式 |
| `CHITCHAT_SYSTEM_PROMPT` | 闲聊模式 |
| `GENERAL_SYSTEM_PROMPT` | 通用知识模式 |

- Prompt 使用 `f-string` 或 `.format()` 插入 `{context}` 等变量
- `RAGChain` 通过 `is_multimodal` 标志选择对应 Prompt
- **禁止在 `chain.py` 或其他文件中硬编码 Prompt**

### 4.6 降级与容错

所有外部调用必须有降级策略：

```python
# 示例: 图片描述生成失败时降级为默认标签
try:
    desc = self._generate_image_description(image_b64, source, page)
except Exception as e:
    logger.warning(f"图片描述生成失败，使用默认标签: {e}")
    desc = f"[图片] 来源: {source}, 第{page}页"
```

降级原则：
- **不因局部失败阻塞主流程**
- **多模态失败不影响纯文本链路**
- **图片描述失败不影响图片入库（使用默认标签）**
- **Embedding 失败可降级到纯 BM25 检索**

## 8. 错误处理规范

### 8.1 日志使用

```python
from utils.logger import get_logger

logger = get_logger("模块名")

logger.info("正常流程信息")
logger.warning("降级或非致命错误")
logger.error("致命错误，需要关注")
logger.debug("调试信息，生产环境不输出")
```

### 8.2 API 错误返回格式

```python
# FastAPI 路由中的错误处理
from fastapi import HTTPException

raise HTTPException(status_code=400, detail="不支持的文件格式")
raise HTTPException(status_code=500, detail="LLM 服务不可用")
```

### 8.3 错误处理矩阵

| 场景 | 处理方式 | 日志级别 |
|------|----------|----------|
| 文件类型不支持 | 返回 400 | WARNING |
| 文件内容为空 | 返回 400 | WARNING |
| LLM API 超时 | 返回 500 | ERROR |
| Embedding 失败 | 降级到 BM25 | WARNING |
| 多模态入库失败 | 不影响纯文本链路 | WARNING |
| 图片描述生成失败 | 降级为默认标签 | WARNING |
| 批量上传部分失败 | 返回成功/失败详情 | WARNING |
| Milvus 连接失败 | 返回 500 | ERROR |

## 9. Git 规范

### 9.1 Commit 消息格式

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>
```

类型：
| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `optimize` | 性能或体验优化 |
| `refactor` | 重构（不改变功能） |
| `docs` | 文档更新 |
| `test` | 测试相关 |
| `chore` | 构建、配置等杂项 |

示例：
```
feat(multimodal): 添加图片描述自动生成
optimize(retriever): 图片入库 chunk_id 去重
docs(architecture): 更新 Prompt 体系说明
```

### 9.2 分支策略

- `main`: 生产分支，保持可用
- `dev`: 开发分支，日常开发在此进行
- `feature/<名称>`: 功能分支，完成后合并到 `dev`

## 10. 常见问题与排查

### 10.1 服务启动失败

```bash
# 查看后端日志
tail -50 logs/backend.log

# 检查端口占用
lsof -i :8000

# 检查 conda 环境
conda info --envs
```

### 10.2 Milvus 数据异常

```bash
# Milvus 数据文件位置
ls -la data/milvus.db

# 如需重置（会丢失所有向量数据）
rm data/milvus.db
./start.sh restart
```

### 10.3 前端页面空白

```bash
# 检查前端是否构建
ls -la web/dist/

# 重新构建
cd frontend && npm run build && cd ..
./start.sh restart
```

### 10.4 LLM 调用超时

- 检查 `config/settings.py` 中 `llm_timeout` 设置（默认 120s）
- 检查网络连通性
- 查看日志中的具体错误信息

### 10.5 Embedding 维度不匹配

- 纯文本链路: 1024 维（bge-large-zh）
- 多模态链路: 4096 维（Qwen3-VL-Embedding）
- 创建 Collection 时维度必须与模型输出维度一致
- **切换模型后必须删除旧 Collection 重建**
