## 1. 现状分析

### 1.1 多模态架构概览

当前系统采用**纯文本 + 多模态双链路**架构：

```
上传文档
    │
    ├── 纯文本链路 (bge-large-zh, 1024d)
    │   └── Milvus Collection: knowledge_base
    │
    └── 多模态链路 (Qwen3-VL-Embedding-8B, 4096d)
        ├── 文本切片 → embed_text()
        ├── 文本切片 → embed_text()
        ├── PDF 图片 → extract_images_from_pdf() → LLM 描述生成 → embed_image()
        ├── Word 图片 → extract_images_from_docx() → LLM 描述生成 → embed_image()
        └── PPT 图片 → extract_images_from_pptx() → LLM 描述生成 → embed_image()
            └── Milvus Collection: knowledge_base_mm
```

两条链路**独立运行、互不干扰**，通过前端模式切换（`text` / `multimodal`）选择使用哪条。

### 1.2 多模态数据流

```
┌─────────────────────────────────────────────────────────────────┐
│ 文档上传 (routes_docs.py)                                        │
│                                                                 │
│  PDF/Word/PPT/Excel → FileParser.parse() → TextChunker.chunk() │
│       │                                                         │
│       ├──→ VectorRetriever.insert_documents()   [纯文本 Collection] │
│       │                                                         │
│       └──→ MultimodalRetriever.insert_documents() [多模态 Collection] │
│            └── PDF 额外: extract_images_from_pdf()              │
│                └── insert_images() [图片切片入库]                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 对话查询 (routes_chat.py → chain.py)                             │
│                                                                 │
│  用户提问 → route_query() → 判断 RAG                             │
│       │                                                         │
│       └── active_chain.search(query)                            │
│           ├── text 模式: VectorRetriever (1024d, bge-large-zh)  │
│           └── multimodal 模式: MultimodalRetriever (4096d, Qwen3-VL) │
│               │                                                 │
│               ├── 向量检索 (COSINE)                              │
│               ├── BM25 文本匹配                                  │
│               └── RRF 融合 → 返回结果                             │
│                   └── has_image + image_url 字段                 │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 图片入库优化

### 3.1 图片内容描述生成（P0）✅ 已实现

**现状问题**：图片入库时 `content` 字段仅为 `[图片] 来源: xxx, 第x页`，无任何语义信息。

**影响**：
- BM25 检索完全无法匹配图片（因为没有可分词的文本）
- 向量检索虽然能通过图片特征匹配，但缺少文字描述会降低跨模态检索精度
- LLM 拿到的上下文中图片只有标签，无法理解图片内容

**优化方案**：入库时用 LLM 生成图片内容描述

```python
# rag/retriever.py — insert_images() 改进
def insert_images(self, images: List[Dict], source: str = "unknown") -> int:
    import hashlib
    data = []
    for img in images:
        page = img.get("page", 0)
        b64 = img.get("b64", "")
        img_path = img.get("path", "")

        # 用多模态 Embedder 向量化图片
        emb = self.embedder.embed_image(b64)

        # 用 LLM 生成图片描述（新增）
        description = self._generate_image_description(b64, source, page)

        chunk_id = hashlib.md5(
            f"{source}_img_p{page}_{img_path}".encode()
        ).hexdigest()[:12]

        data.append({
            "chunk_id": chunk_id,
            "doc_id": "",
            "filename": source,
            "content": description,  # 从标签改为实际描述
            "chunk_index": -1,
            "page_number": page,
            "has_image": True,
            "image_url": img_path,
            "embedding": emb,
        })

    self._client.insert(self.collection_name, data)
    self._client.flush(self.collection_name)
    return len(data)

def _generate_image_description(self, image_b64: str, source: str, page: int) -> str:
    """用 LLM 生成图片内容描述"""
    try:
        # 构建多模态消息（mimo-v2.5 支持 vision）
        messages = [
            {"role": "user", "content": [
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{image_b64}"}},
                {"type": "text", "text": f"请用中文简要描述这张图片的内容，这是文档《{source}》第{page}页的图片。只描述图片内容，不要添加额外说明。"}
            ]}
        ]
        # 调用 LLM（同步 httpx，匹配项目模式）
        headers = {"Content-Type": "application/json", "Authorization": f"Bearer {settings.llm_api_key}"}
        payload = {"model": settings.llm_model_name, "messages": messages, "max_tokens": 200, "temperature": 0}
        with httpx.Client(timeout=30) as client:
            resp = client.post(settings.llm_api_url, json=payload, headers=headers)
            resp.raise_for_status()
            desc = resp.json()["choices"][0]["message"]["content"].strip()
            logger.info(f"图片描述生成: {source} 第{page}页 -> {desc[:50]}...")
            return f"[图片] {desc}（来源: {source}, 第{page}页）"
    except Exception as e:
        logger.warning(f"图片描述生成失败，使用默认标签: {e}")
        return f"[图片] 来源: {source}, 第{page}页"
```

**预期收益**：
- BM25 可以通过描述文本匹配图片
- 向量检索因描述文本辅助而精度提升
- LLM 上下文中图片有了可理解的内容

**注意事项**：
- LLM 调用会增加入库时间（每张图 ~2-5 秒）
- 需要 LLM 支持 vision 能力（mimo-v2.5 已支持）
- 失败时降级为原有标签，不影响入库流程

## 4. 图片检索优化

### 4.1 图片向量检索增强（P1）

**现状问题**：当前图片检索仅依赖 Qwen3-VL 的向量相似度，没有利用图片的上下文信息（如所在页面的文本、图片描述等）。

**优化方案**：多路检索融合

```
用户查询
    │
    ├── 1. 文本向量检索（embed_text(query) vs 文本切片向量）
    ├── 2. 图片向量检索（embed_text(query) vs 图片向量）
    ├── 3. BM25 文本匹配（jieba 分词）
    └── 4. 描述文本向量检索（embed_text(query) vs 图片描述向量）
         │
         ▼
    RRF 融合 → 最终结果
```

**关键改动**：图片入库时同时存储描述文本的向量（可以用 bge-large-zh 编码描述文本），检索时两路向量都参与 RRF 融合。

---

### 4.2 跨模态检索优化（P1）

**现状问题**：纯文本查询和图片查询使用同一个多模态 Embedding 模型，但 4096 维向量的检索效率低于 1024 维。

**优化方案**：

- 对于纯文本查询，可以同时用 bge-large-zh (1024d) 检索文本部分，用 Qwen3-VL (4096d) 检索图片部分
- 两路结果通过 RRF 融合
- 这样既保留了文本检索的高精度，又增加了图片检索能力

```python
class HybridMultimodalRetriever:
    """混合多模态检索器：文本用 bge-large-zh，图片用 Qwen3-VL"""

    def __init__(self):
        self.text_embedder = EmbeddingClient()      # bge-large-zh, 1024d
        self.mm_embedder = MultimodalEmbedder()     # Qwen3-VL, 4096d

    def search(self, query, top_k=5):
        # 文本向量检索（1024d Collection）
        text_vec = self.text_embedder.embed_query(query)
        text_results = self._search_collection("knowledge_base", text_vec, top_k)

        # 多模态向量检索（4096d Collection）
        mm_vec = self.mm_embedder.embed_text(query)
        mm_results = self._search_collection("knowledge_base_mm", mm_vec, top_k)

        # RRF 融合
        return self._rrf_fusion(text_results, mm_results)[:top_k]
```

## 7. 文档格式扩展

### 7.1 Word/PPT 图片提取（P1）✅ 已实现

**现状问题**：`file_parser.py` 的 `_parse_docx()` 和 `_parse_pptx()` 只提取文字，忽略内嵌图片。

**优化方案**：

```python
# utils/file_parser.py — 新增方法
@staticmethod
def extract_images_from_docx(file_path: str, output_dir: str) -> List[Dict]:
    """从 Word 文档中提取图片"""
    from docx import Document
    import base64, os
    doc = Document(file_path)
    os.makedirs(output_dir, exist_ok=True)
    images = []
    for rel in doc.part.rels.values():
        if "image" in rel.reltype:
            img_data = rel.target_part.blob
            ext = rel.target_part.content_type.split("/")[-1]
            img_name = f"img_{rel.rId}.{ext}"
            img_path = os.path.join(output_dir, img_name)
            with open(img_path, "wb") as f:
                f.write(img_data)
            images.append({
                "page": 0,  # Word 无精确页码
                "path": img_path,
                "b64": base64.b64encode(img_data).decode(),
            })
    return images

@staticmethod
def extract_images_from_pptx(file_path: str, output_dir: str) -> List[Dict]:
    """从 PPT 中提取图片"""
    from pptx import Presentation
    import base64, os
    prs = Presentation(file_path)
    os.makedirs(output_dir, exist_ok=True)
    images = []
    for slide_idx, slide in enumerate(prs.slides, 1):
        for shape in slide.shapes:
            if shape.shape_type == 13:  # MSO_SHAPE_TYPE.PICTURE
                img_data = shape.image.blob
                ext = shape.image.content_type.split("/")[-1]
                img_name = f"img_s{slide_idx}_{shape.shape_id}.{ext}"
                img_path = os.path.join(output_dir, img_name)
                with open(img_path, "wb") as f:
                    f.write(img_data)
                images.append({
                    "page": slide_idx,
                    "path": img_path,
                    "b64": base64.b64encode(img_data).decode(),
                })
    return images
```

## 8. 图文混排回答

### 8.1 LLM 图文混排标记（P2）

**现状问题**：LLM 回答是纯文本，图片只是附在参考来源区域，无法在回答正文中展示。

**优化方案**：让 LLM 在回答中标注图片引用，后端解析后替换为实际图片 URL。

#### Prompt 引导

```python
# rag/prompt_template.py — 多模态专用 Prompt
MULTIMODAL_SYSTEM_PROMPT = """你是一个专业的知识问答助手，支持图文混合回答。

回答规则：
1. 严格基于参考资料回答
2. 当参考资料中包含图片时，可以在回答中引用图片，格式为 ![参考X]
3. 图片引用应放在相关内容附近，帮助用户理解
4. 其他规则同标准 SYSTEM_PROMPT

参考资料：
{context}
"""
```

#### 后端解析

```python
# rag/chain.py — 回答后处理
def _postprocess_answer(self, answer: str, docs: List[Dict]) -> str:
    """将 ![参考X] 标记替换为实际图片 URL"""
    import re
    for i, doc in enumerate(docs, 1):
        if doc.get("has_image") and doc.get("image_url"):
            answer = answer.replace(f"![参考{i}]", f"![参考{i}]({doc['image_url']})")
    return answer
```

**注意事项**：
- 需要 LLM 理解并遵循图文混排格式
- 前端 Markdown 渲染器（marked.js）已支持图片语法，无需额外改动
- 可能需要多次调优 Prompt 才能让 LLM 正确使用图片引用
