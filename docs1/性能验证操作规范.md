# 性能验证操作规范

> **用途**：当代码评估怀疑存在性能问题（或修复后需要验证效果）时，按本规范执行验证，确保结论可信、可复现、可回归。
> **适用场景**：性能瓶颈验证、优化前后对比、规模化压力测试。
> **不适用**：功能正确性测试（那是单元测试的职责）。

---

## 核心原则（先记住这五条）

| 原则 | 含义 | 反面案例 |
|------|------|---------|
| **1. 先证明问题存在，再谈验证** | 用 grep/读代码确认可疑代码路径真实存在，不基于假设写测试 | 没读代码就猜"这里可能慢"然后测半天 |
| **2. 用真实环境，不 mock 核心依赖** | 性能测试必须用项目真实的依赖版本和运行环境 | 用 mock 的 jieba / mock 的数据库,测出来的数字毫无意义 |
| **3. 控制变量，单一差异** | 对照组和实验组只有"待验证的那一个差异"，其余全相同 | A 组用真数据 B 组用合成数据,然后对比 |
| **4. 数据自己说话，禁止估算** | 所有结论数字来自测试输出，不来自"我觉得""大概""通常" | 报告里写"应该能提升 5-10 倍" |
| **5. 写下来的才是规范** | 测试脚本必须留存，路径固定，可重复执行 | 测完即删，下次想回归找不到了 |

---

## 标准操作流程（六步法）

### 第 1 步：定位与确认（5 分钟）

**目标**：证明"被怀疑的代码路径"真实存在，且确实是性能热点。

**操作**：

```bash
# 1. 用 grep 定位代码
grep -rn "可疑函数名\|可疑模式" --include="*.py" .

# 2. 读完整函数体（不是片段），确认逻辑
```

**检查清单**：
- [ ] 这个函数/循环确实在热路径上吗？（被高频调用 vs 启动时调用一次）
- [ ] 昂贵操作（I/O、分词、建索引、网络）真的在循环里 / 函数内吗？
- [ ] 有没有可能它已经被缓存了，只是我没看到？

**产出**：一段引文 + 文件:行号，证明问题位置。**没有产出就停止，不要往下做。**

**示例**（本次 BM25 验证的产出）：
```
retriever.py:192-206 _bm25_search() 内部：
  line 203: corpus_tokens = [list(jieba.cut(...)) for doc in self._bm25_docs]  ← 每次查询都重新分词
  line 206: bm25 = BM25Okapi(corpus_tokens)  ← 每次查询都重建实例
确认：两个 O(N) 操作在查询热路径内，不是在缓存函数 _update_bm25_cache 里。
```

---

### 第 2 步：设计对照实验（10 分钟）

**目标**：明确"对照组（现状）"和"实验组（优化后）"的差异边界。

**必须回答三个问题**：

1. **变量是什么**？只能有一个。
   - 本次：是否缓存 BM25 实例（对照组=每次重建，实验组=复用）
2. **控制了什么**？
   - 本次：查询语句相同、切片内容相同、分词算法相同、Python 环境相同
3. **衡量指标是什么**？
   - 本次：单次查询延迟（ms/查询）

**对照设计模板**：

```python
def bench_current_impl(...):
    """对照组：当前生产代码逻辑"""
    # 完全复刻生产代码的可疑路径
    ...

def bench_optimized_impl(...):
    """实验组：优化后逻辑"""
    # 只改变那一个变量
    ...

# 两者用完全相同的输入
```

**禁忌**：
- ❌ 实验组顺便优化了别的东西（比如换了分词器）→ 无法归因
- ❌ 用不同的数据集 → 结果不可比
- ❌ 改了输入规模又改了实现 → 混淆因子

---

### 第 3 步：准备测试数据（10 分钟）

**优先级**（从高到低）：

| 优先级 | 数据来源 | 何时用 |
|--------|---------|--------|
| 🥇 最佳 | 生产环境真实数据 | 任何时候都优先 |
| 🥈 次选 | 合成数据，但参数贴近真实 | 拿不到生产数据时 |
| 🥉 最次 | 纯随机/合成数据 | 仅做粗略验证 |

**"贴近真实"的标准**——必须校准这些参数：

- **数据规模**：用真实测量值（如本次 53 切片是实际查 Milvus 得到的）
- **单条数据尺寸**：用真实分布（本次 400 字符/段匹配实际切片）
- **数据特征**：语言、编码、结构要一致（本次用中英混合，匹配知识库内容）

**本次 BM25 测试的数据生成**：

```python
def make_corpus(n_chunks, chars_per_chunk=400):
    """生成模拟切片（参数校准：400 字符匹配项目 chunk_size=500 扣除重叠后的实际尺寸）"""
    sample_sentences = [...真实知识库风格的句子...]
    corpus = []
    for i in range(n_chunks):
        text = ""
        while len(text) < chars_per_chunk:
            text += sample_sentences[i % len(sample_sentences)] + "。"
        corpus.append(text[:chars_per_chunk])
    return corpus
```

**坦诚标注**：本次测试用的是 🥈 合成数据（次选），因为没从 Milvus 拉真实切片。**这是本规范的已知局限**，应在报告里如实说明。

---

### 第 4 步：编写基准脚本（15 分钟）

**脚本结构标准**：

```python
"""
[问题名称] 性能验证脚本
验证点：
1. [对照组的瓶颈描述]
2. [实验组的优化方案]
3. 在 [N 个规模梯度] 下量化差异
"""
import time
# ... 依赖导入 ...

# ========== 数据准备 ==========
def make_test_data(...): ...

# ========== 对照实验 ==========
def bench_current(...):
    """对照组：复刻当前生产代码"""
    # 关键：完整复刻，不要简化
    ...

def bench_optimized(...):
    """实验组：只改一个变量"""
    ...

# ========== 执行与统计 ==========
def main():
    # 多规模梯度
    scales = [(53, "当前规模"), (500, "中等"), (2000, "较大"), (5000, "大规模")]
    # 多轮取平均
    ROUNDS = 5
    # warmup（见下方"统计规范"）
    ...

if __name__ == "__main__":
    main()
```

**统计规范**：

1. **必须有 warmup**——首轮受 JIT/词典加载/缓存未命中影响，必须丢弃：

```python
# warmup：跑 2-3 轮但不计入统计
for _ in range(2):
    bench_current(data, queries)

# 正式测量：N 轮
times = []
for _ in range(N):
    t0 = time.perf_counter()  # ← 用 perf_counter，不用 time.time()
    ...
    times.append(elapsed)
```

2. **报告 avg + min（或 avg ± std）**，不要只报单次值：

```python
avg = sum(times) / len(times)
mn = min(times)
print(f"avg={avg:.2f}ms  min={mn:.2f}ms")
```

3. **用 `time.perf_counter()` 不用 `time.time()`**——前者单调时钟，不受系统时间调整影响。

4. **规模梯度至少 3 档**，最好包含"当前规模"和"10x/100x 未来规模"。

**本次测试的统计缺陷（坦诚）**：
- ❌ 没做 warmup——首轮可能受 jieba 词典加载影响（基准输出末尾的 "Loading model cost 1.467 seconds" 就是证据）
- ✅ 做了多轮取平均（5 轮）
- ✅ 用了 perf_counter
- ✅ 做了 4 档规模梯度

### 第 5 步：在真实环境执行（5 分钟）

**必须满足**：

1. **用项目的运行环境**，不是系统默认 Python：

```bash
# ✅ 正确：激活项目 conda 环境
wsl.exe -e bash --norc --noprofile -c \
  "source /home/yhwz/miniconda3/etc/profile.d/conda.sh && \
   conda activate kbqa && \
   python docs1/_bm25_bench.py"

# ❌ 错误：用系统 python（依赖版本可能不同）
python docs1/_bm25_bench.py
```

2. **关闭无关进程**——性能测试时不要同时跑其他吃 CPU/内存的任务。

3. **记录环境元信息**——报告里要写清 Python 版本、关键依赖版本、机器配置。

**本次的环境记录**：
```
环境：WSL Ubuntu / conda env "kbqa" / Python 3.12.13
关键依赖：jieba 0.42.1, rank_bm25 0.2.2
```

---

### 第 6 步：产出报告与结论（10 分钟）

**报告必须包含**：

1. **问题描述**（来自第 1 步）：代码位置 + 为什么是热点
2. **实验设计**（来自第 2 步）：变量、控制、指标
3. **数据表格**：规模 × 实现方式 × 延迟
4. **加速比计算**：`speedup = 当前耗时 / 优化后耗时`
5. **结论**：用数字说话，带条件
6. **已知局限**：诚实标注（合成数据？没测并发？没 warmup？）

**结论写作规范**：

✅ 好的结论：
> 当前 53 切片规模下，每次 BM25 查询耗时约 99ms；优化后降至 0.13ms，加速 783 倍。到 5000 切片规模，当前实现需 4.2 秒，优化后 6.5ms。

❌ 不好的结论：
> BM25 性能有显著提升，优化效果明显。
> （空话，没数字）

❌ 更不好的结论：
> 预计优化后性能能提升数百倍。
> （"预计"——没测就别下结论）

## 测试脚本管理规范

### 存放位置

```
项目根/
├── tests/
│   └── perf/                      # 性能测试专用目录
│       ├── README.md              # 说明每个脚本的用途
│       ├── bench_bm25_rebuild.py  # BM25 重建实例问题
│       ├── bench_retriever.py     # 检索器整体延迟
│       └── bench_chunker.py       # 切片器吞吐
```

### 命名规范

```
bench_<被测模块>_<验证点>.py

例：
bench_bm25_rebuild.py      ← BM25 实例重建问题
bench_retriever_search.py  ← 检索器 search 整体延迟
bench_chunker_throughput.py ← 切片器吞吐量
```

### 生命周期

| 阶段 | 动作 |
|------|------|
| 创建时 | 放入 `tests/perf/`，写清 docstring（验证什么、怎么跑、预期结果） |
| 验证完成后 | **保留**，不要删——用于回归 |
| 优化合并后 | 作为回归测试，定期跑，防止性能回退 |

**本次 BM25 测试的违规点**：脚本跑完就删了（`rm _bm25_bench.py`），违反了"保留可回归"原则。**正确做法是移到 `tests/perf/bench_bm25_rebuild.py`**。

## 何时必须做性能验证（触发条件）

不是所有怀疑都要走完整流程。按强度分三档：

### 🔴 必须完整验证（走六步法）

- 评估报告里要写进 P0/P1 的性能结论
- 优化前后对比，要在文档里给数字
- 影响核心链路（检索、生成、入库）的性能改动

### 🟡 轻量验证（grep + 读代码即可，可不跑测试）

- 怀疑某处慢，但不确定是不是热点
- 低频路径（如配置保存、日志写入）
- 只是想确认"有没有明显问题"

### 🟢 无需验证

- 功能正确性问题（用单元测试）
- 代码风格、命名、结构问题（code review）
- 纯文档问题

**本次 BM25 属于 🔴**：要写进评估报告的 P0 结论，且影响核心检索链路，所以走了完整六步法。

## 常见陷阱清单（排雷手册）

| 陷阱 | 现象 | 规避方法 |
|------|------|---------|
| **没 warmup** | 首轮数据异常高 | 丢弃前 2-3 轮 |
| **用 `time.time()`** | 系统时间跳变导致负数或异常 | 用 `time.perf_counter()` |
| **mock 了核心依赖** | 测出"理想"数字,生产环境对不上 | 性能测试禁止 mock 被测的核心组件 |
| **单次测量** | 偶发抖动当结论 | 至少 5 轮取平均 |
| **规模只有一档** | 不知道规模化表现 | 至少 3 档,含当前规模 + 未来规模 |
| **对照组有多个差异** | 无法归因到具体改动 | 严格控制单一变量 |
| **GC 抖动** | Python GC 间歇性卡顿 | 可用 `gc.disable()`（高级,可选） |
| **测试数据不真实** | 合成数据太"干净" | 校准规模、尺寸、特征三个参数 |
| **测完删脚本** | 无法回归 | 存到 `tests/perf/` |
| **结论用"预计/大概"** | 主观估计当数据 | 所有数字来自测试输出 |

## 附：一页纸速查卡

```
性能验证六步法：

1. 定位 → grep + 读代码，确认热点真实存在
2. 设计 → 单一变量，明确对照/实验组差异
3. 数据 → 真实数据优先，合成数据要校准规模+尺寸+特征
4. 脚本 → warmup + 多轮平均 + perf_counter + 多规模梯度
5. 执行 → 用项目真实环境，记录元信息
6. 报告 → 数字说话，标注局限，脚本留存可回归

五条铁律：
① 先证明存在再验证
② 不 mock 核心依赖
③ 单一变量
④ 禁止估算
⑤ 脚本必须留存
```
